# `ai/` — 컬럼 하나가 없어서 태어난 20시간짜리 괴물

### 스마트쉼터 탐지 파이프라인: 공공데이터 거버넌스 공백이 낳은 엔지니어링 실패 보고서

> *"국토교통부 CSV 파일에 `is_smart_shelter` 컬럼 하나만 있었어도, 이 폴더는 존재할 이유가 없었다."*

이 폴더는 스마트쉼터 지도 서비스([smartshelter.co.kr](https://smartshelter.co.kr))의 AI 파이프라인이다. 정확히는, **있어야 할 데이터가 없어서 한 사람이 GPU도 없는 노트북으로 20시간 동안 벌인 삽질의 화석**이다. 이 README는 사용 설명서가 아니라, 그 삽질이 왜 필연이었는지에 대한 증언이다.

|  | 있어야 했던 것 | 대신 우리가 만들어야 했던 것 |
|---|---|---|
| 데이터 | 컬럼 1개 (`is_smart_shelter: Y/N`) | 이미지 234장 → 2,340장 (Albumentations 10x 증강) |
| 조회 방법 | CSV 컬럼 필터링 한 줄 (`is_smart_shelter == 'Y'`) | YOLOv8n 모델 Fine-tuning (i3 내장그래픽, 20시간) |
| 검증 방법 | 이미 행정에 존재하는 설치 대장 | 카카오맵 + 네이버맵 로드뷰 이중 크롤링 |
| 신뢰성 확보 | 공식 데이터라는 사실 자체 | 봇 차단 피하려 `time.sleep()` 떡칠, 재시도 로직 |
| 총 소요 코드량 | 0줄 | 239줄의 스파게티 |

---

## 1. 이 괴물은 왜 태어났는가

국토교통부는 「전국 버스정류장 위치정보」라는 이름으로 전국 정류장의 좌표 데이터를 공공데이터포털에 공개한다. 정류장 ID, 정류장명, 위도, 경도 — 여기까지는 훌륭하다.

그런데 이 데이터에는 그 정류장이 **일반 정류장인지, 냉난방·공기청정 시스템이 갖춰진 '스마트쉼터'인지**를 구분하는 컬럼이 존재하지 않는다. 설치 주체(지자체·행안부·각 구청 시설관리공단)는 이미 이 정보를 알고 있다. 어딘가의 행정 시스템에는 스마트쉼터 설치 목록이 엑셀로든 뭐로든 존재할 것이다. 하지만 그것이 공공데이터로 개방되어 있지 않다.

그 결과, 이 정보를 필요로 하는 민간 개발자에게 남은 선택지는 단 하나였다.

> **좌표값만 가지고, 로드뷰를 열어서, 사람이 하나하나 눈으로 확인하는 일을, AI에게 대신 시킨다.**

컬럼 하나의 부재가, 이미지 인식 모델 학습과 브라우저 자동화 크롤링이라는 완전히 별개의 공학 문제 두 개를 만들어냈다. 이것이 `ai/` 폴더의 모든 코드가 존재하는 유일한 이유다.

## 2. 기형적인 시스템 아키텍처

코드는 지저분하지만, 이 지저분함에도 나름의 레이어가 있다. 실제로 존재해서는 안 됐을 파이프라인을, 존재해버렸으니 정직하게 그려본다.

```
─────────────────────────────────────────────────────────────────────
   [Layer 0] 데이터 원천의 결핍                                        
   국토교통부_전국 버스정류장 위치정보.csv                              
   (좌표는 있음. 정답(Y/N)은 없음. cp949 인코딩이라는 덤도 있음.)         
───────────────────────────────┬─────────────────────────────────────
                               ▼
─────────────────────────────────────────────────────────────────────
   [Layer 1] Synthetic Ground-Truth Augmentation                      
   Albumentations 파이프라인                                          
   234장 raw → 2,340장 augmented                                      
   (OpticalDistortion / GaussNoise / ShiftScaleRotate)                 
   "로드뷰 카메라의 열화·왜곡을 인위적으로 재현하여 실전에 대비"           
───────────────────────────────┬─────────────────────────────────────
                               ▼
─────────────────────────────────────────────────────────────────────
   [Layer 2] Inference Engine                                         
   YOLOv8n Fine-tuning, epochs=50                                     
   학습 장비: GPU 없음. i3 내장그래픽. 소요시간 약 20시간.               
───────────────────────────────┬─────────────────────────────────────
                               ▼
─────────────────────────────────────────────────────────────────────
   [Layer 3] Visual Acquisition Layer (RPA)                           
   Selenium 기반 이중 진입 크롤러                                        
   ├─ 카카오맵 로드뷰 링크 진입 → 스크린샷                               
   └─ 네이버맵 진입 → 거리뷰 버튼 클릭 → 화면 중앙 강제 클릭              
      → 거리뷰 내부 진입 → 스크린샷                                     
   (전국 정류장 수 × 2개 지도 서비스 × 왕복 요청)                         
───────────────────────────────┬─────────────────────────────────────
                               ▼
─────────────────────────────────────────────────────────────────────
   [Layer 4] Detection & Persistence                                 
   캡처 이미지 → YOLO 추론 → 박스 존재 여부 판정                        
   → data.txt (現) → smartshelter.co.kr DB INSERT (向, TODO)          
─────────────────────────────────────────────────────────────────────
```

엔터프라이즈 시스템처럼 그려놨지만, Layer 3의 실체는 **브라우저 창의 정중앙 픽셀 좌표를 계산해서 마우스로 그냥 찍어버리는 매크로**다. 이게 "거리뷰 진입 로직"의 전부다.

## 3. 실행 방법

이 파이프라인은 "학습은 1회, 스캔은 반복" 구조다. 데이터 없이는 아무것도 시작되지 않으니, 순서대로 준비물부터 채워 넣어야 한다.

### 3.1 사전 준비

1. **스마트쉼터 사진 확보**
   `ai/dataset/raw/images/` 폴더에 스마트쉼터가 찍힌 사진을 넣는다 (`.jpg` / `.png`).

2. **바운딩 박스 라벨링**
   [LabelImg](https://github.com/HumanSignal/labelImg) 등으로 위 이미지에 직접 박스를 치고, **YOLO 포맷**으로 저장한 `.txt` 파일을 `ai/dataset/raw/labels/`에 넣는다. 이미지와 라벨 파일명(확장자 제외)이 정확히 일치해야 한다 (`정류장1.jpg` ↔ `정류장1.txt`).

3. **버스정류장 좌표 CSV 다운로드**
   [공공데이터포털 - 전국 버스정류장 위치정보](https://www.data.go.kr/data/15067528/fileData.do)에서 CSV를 다운로드해 `ai/` 폴더 바로 아래에 둔다. (인코딩은 `cp949`)

필요한 라이브러리는 `opencv-python`, `albumentations`, `ultralytics`, `selenium`, `webdriver-manager` 등이다.

### 3.2 모델 학습 (최초 1회, 약 20시간)

`search_shelter.py`의 `__main__` 블록에서 주석(`'''...'''`) 처리된 아래 구간을 해제하고 실행한다.

```python
pipeline.augment_data(augment_factor=10)   # 234장 → 2,340장 증강
pipeline.train_model(epochs=50)            # YOLOv8n 학습
```

- 1단계에서 준비한 이미지·라벨을 기반으로 데이터 증강이 먼저 돌고, 그 결과물로 YOLOv8n을 파인튜닝한다.
- 학습이 끝나면 `runs/detect/shelter_detector/weights/best.pt`에 가중치가 저장된다.
- **학습은 데이터셋이 바뀌지 않는 한 반복할 필요가 없다.** 한 번 끝났으면 다시 주석 처리해서 매 실행마다 20시간을 태우는 일을 막는다.

### 3.3 전국 정류장 스캔

```bash
python search_shelter.py
```

- CSV를 한 줄씩 읽어(헤더 제외) 좌표마다 카카오맵·네이버맵 로드뷰를 캡처하고, 학습된 모델로 스마트쉼터 여부를 판별한다.
- 스마트쉼터로 판별된 정류장은 `ai/data.txt`에 `정류장코드,정류장ID,위도,경도` 형식으로 한 줄씩 누적 기록된다.

## 4. 아름다운 실패의 증거들

대부분의 경우였다면 코드 리뷰 단계에서 즉시 반려되었을 것들이다. 그런데도 이것들이 **데이터가 없는 1인 개발자가 낼 수 있는 최선의 결과물**이었다는 걸 여기서 변호한다.

- **화면 중앙 강제 클릭 매크로**
  네이버 지도 거리뷰에 진입하는 "정식 API"는 없다. 그래서 브라우저 창의 가로세로 절반 지점을 계산해서 `ActionChains`로 그냥 눌러버린다. 실패하면 `body` 엘리먼트를 한 번 거쳐서 재시도한다. 이게 네이버맵 진입 로직의 전부이며, 이보다 우아한 방법은 존재하지 않았다.

- **절대경로 XPath 하드코딩**
  `//*[@id="app-layout"]/div[3]/div/div[3]/div[3]/div[1]/div[2]/button[4]` — 이 좌표는 네이버 프론트엔드가 CSS 클래스 하나만 바꿔도 통째로 죽는다. 실제로 개발 도중 몇 번이나 이 이유로 크롤러가 멈췄다. 그런데도 이렇게 짤 수밖에 없었던 이유는 단순하다: **공식 스트리트뷰 조회 API가 없기 때문**이다. 화면을 흉내 내는 것 외에는 방법이 없었다.

- **렌즈 왜곡을 일부러 만들어내는 데이터 증강**
  234장의 원본 이미지를 `OpticalDistortion`, `GaussNoise`, `ShiftScaleRotate`로 두들겨서 2,340장으로 부풀렸다. 로드뷰 카메라 특유의 화질 저하와 왜곡을 학습 단계에서부터 흉내 낸 것이다. 데이터 234장은 절대적으로 부족했고, 이걸 메꿀 수 있는 건 통계가 아니라 트릭뿐이었다.

- **카카오 + 네이버 이중 크롤링**
  둘 중 어느 한쪽도 전국 정류장 로드뷰를 완벽하게 커버하지 못한다. 그래서 한 정류장당 지도를 두 번 열고, 브라우저를 두 번 붙잡고, 스크린샷을 두 장 찍는다. 비효율의 극치이지만, 이것이 **단일 실패 지점(SPOF)을 줄이는 유일한 방법**이었다.

- **`time.sleep()` 떡칠**
  4.5초, 3초, 1.5초, 0.5초… 포털사의 봇 탐지를 피하고 렌더링을 기다리기 위해 코드 곳곳에 흩뿌려진 매직 넘버들이다. 우아한 `WebDriverWait` 대신 이런 무식한 대기가 들어간 이유는, **비공식 크롤링에는 "안정적인 신호" 자체가 없었기 때문**이다.

- **GPU 없는 노트북에서 20시간**
  i3 내장 그래픽으로 YOLOv8n을 50 epoch 학습시키는 데 20시간이 걸렸다. 클라우드 GPU를 빌릴 예산도, 그럴 만한 여유도 없는 1인 개발자에게 이건 사양이 아니라 **유일한 옵션**이었다.

이 코드의 모든 못생긴 줄은, 정답을 몰라서 눈으로 직접 확인해야 했던 사람의 마지막 발버둥을 기계에게 대신 시킨 흔적이다.

## 5. 결론 및 제언

이 코드는 실패했다. 실행할 때마다 XPath 하나에 죽고, 봇 차단에 걸리고, 20시간을 태워 학습시킨 모델의 정확도는 여전히 불안정하다. 자랑스럽게 내세울 만한 엔지니어링은 아니다.

하지만 이 실패에는 아주 명확한 원인과, 아주 간단한 해법이 있다.

이 폴더에 있는 데이터 증강 로직도, YOLO 학습도, 셀레니움 매크로도 — 국토교통부(또는 각 지자체)가 이미 알고 있는 정보, 즉 **"이 정류장은 스마트쉼터인가"라는 참/거짓 값 하나**를 공공데이터로 열었다면 단 한 줄도 필요하지 않았다.

민간은 없는 데이터를 만들어내기 위해 AI를 학습시키고, 지도 서비스를 크롤링하고, 20시간짜리 컴퓨팅을 태운다. 정부는 이미 갖고 있는 값을 컬럼 하나 추가하는 것으로 끝낼 수 있다.

**이 폴더는 그 비대칭에 대한 증거물로 남는다.**
